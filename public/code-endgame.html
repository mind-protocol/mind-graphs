2 · Un exemple, de bout en bout

Le rapport Basel : comment une phrase devient un effet prouvé dans le monde — sans écrire un service, sans toucher un fichier.

①

L’intention brute

ambiguë · non exécutable
« Envoie le rapport à Basel. »

Qui autorise ? Combien de fois ? Qu’est-ce qui prouve l’envoi ? Que faire si Telegram ne répond pas ? Une intention sans ses preuves n’est pas un programme : c’est une ambiguïté que la boucle conversationnelle doit d’abord résoudre.

②

L’intention compilée

proof-carrying intention
Goal
Basel a reçu le rapport validé, exactement une fois.
CurrentState
Un rapport validé pour Basel est disponible ; rien n’a été envoyé.
Constraints
Proposer l’envoi d’abord ; n’envoyer qu’après approbation.
ForbiddenEffects
Jamais de double envoi ; jamais d’envoi sans approbation.
Delegation
NLR approuve ; le Citizen exécute.
SuccessCriteria
Telegram renvoie un message_id.
FailurePolicy
Résultat ambigu → vérifier avant tout réessai.
Budget
1 envoi, fenêtre de 24 h, attention de NLR : une seule demande.
Reversibility
Non annulable → d’où la porte d’approbation en amont.
③

Le routage

quel régime d’exécution ?

Régime 1 · interprétation directe

telegram.send_message existe, la délégation est active : le runtime orchestre le sous-graphe. Aucun code généré.

Régime 2 · compilation JIT

Si la capacité manquait : génération → tests synthétiques → sandbox → vérification → exécution. Code jetable après usage.

Régime 3 · cristallisation

Si ce plan revenait chaque semaine : structure stable + tests accumulés → Capability permanente « envoi de rapport approuvé ».

④

L’exécution observée

chaque fait est un Moment daté
  1. t₀ Rapport validé détecté → proposition d’envoi à NLR.
  2. t₁ NLR approuve → Moment · Approbation (signé, délégation consommée).
  3. t₂ Envoi via l’outbox — idempotent : un seul tir possible.
  4. t₃ Telegram renvoie message_id 4271 → Moment · Preuve de réussite.
  5. t₃′ Réponse ambiguë ? → vérification de l’état réel avant tout réessai (FailurePolicy).
⑤

La traversée « pourquoi »

toujours disponible

Six mois plus tard : « pourquoi ce message a-t-il été envoyé ? » Le système répond par une traversée, jamais par une supposition :

  1. message Telegram 4271 effet observé
  2. telegram.send_message capacité
  3. SuccessCriteria = message_id test
  4. approbation du 12 mars décision
  5. intention « rapport Basel » proof-carrying
  6. NLR autorité humaine

Un comportement sans cette chaîne n’est pas une curiosité : c’est un défaut.
